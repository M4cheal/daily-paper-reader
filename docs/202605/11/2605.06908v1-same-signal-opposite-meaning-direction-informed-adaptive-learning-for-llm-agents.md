## 摘要
大语言模型（LLM）智能体的自适应测试时计算旨在仅在能够提升性能时调用额外的计算资源。现有方法通常使用基于置信度、不确定性或难度的门控机制，并假设从门控信号到计算需求再到计算价值之间存在固定的方向。这使得门控成为了一个效用校准问题：门控信号应与“额外计算是否比基础策略更能改善最终结果”保持一致。我们证明了这种一致性是不稳定的：同一信号在一种设置下预示着展开（rollout）获益，而在另一种设置下则预示着展开受损，即使在任务固定的情况下，这种反转也会随环境和骨干网络的变化而发生。因此，方向错误的门控可能会因为精确选择了有害状态而导致性能恶化。这种反转反映了计算需求与计算适用性之间更深层次的区别：高不确定性信号可能表示决策困难的状态（此时展开有助于比较备选方案），也可能表示不适合干预的状态（此时当前上下文不支持基于展开的有效改进）。在这种双源模型下，固定方向的门控在异构设置中是不可靠的。为解决这一问题，我们提出了 DIAL（方向感知自适应学习），这是一种通过信号无关的反事实探索训练的稀疏门控，用于学习每个（环境，骨干网络）组合下状态特征的效用方向。在六个环境和三个骨干网络上的实验表明，与固定方向的基准方法相比，DIAL 实现了更强的整体成功率与成本权衡。

## Abstract
Adaptive test-time compute for LLM agents aims to invoke extra computation only when it improves performance. Existing methods typically use confidence-, uncertainty-, or difficulty-based gates, assuming a fixed direction from the gating signal through compute need to the value of computation. This makes gating a utility-calibration problem: gating signals should align with whether extra computation improves the final outcome over the base policy. We show that this alignment is unstable: the same signal predicts rollout benefit in one setting and rollout harm in another, with reversals across environments and backbones even when the task is fixed. Wrong-direction gates can therefore worsen performance by precisely selecting harmful states. This reversal reflects a deeper distinction between compute need and compute suitability: a high uncertainty signal may indicate decision-difficult states where rollouts help compare alternatives, or intervention-unsuitable states where the current context does not support useful rollout-based improvement. Under this two-source model, fixed-direction gates are unreliable across heterogeneous settings. To address this, we propose DIAL (Direction-Informed Adaptive Learning), a sparse gate trained from signal-agnostic counterfactual exploration to learn the utility direction of state features per (environment, backbone). Across six environments and three backbones, DIAL yields a stronger overall success-cost trade-off than fixed-direction baselines.

### 1. 核心问题与研究动机
*   **研究背景**：为了提升 LLM 智能体在复杂任务中的表现，研究者通常采用“测试时计算”（Test-time Compute），即在推理阶段通过多次采样、Rollout（展开）或搜索来优化决策。
*   **核心问题**：现有的自适应计算方法（门控机制）普遍遵循**“固定方向假设”**，即认为更高的不确定性（熵）、更低的置信度或更高的任务难度信号，必然意味着需要更多的额外计算。
*   **关键发现**：作者通过实验证明，这种假设在异构环境下是不稳定的。同一信号（如 Token 熵）在某些环境下与计算收益正相关，但在另一些环境下（或换一个模型骨干）则呈负相关。如果门控方向错误，系统会精确地在“有害状态”下调用额外计算，反而导致性能下降。

### 2. 方法论：DIAL (Direction-Informed Adaptive Learning)
DIAL 的核心思想是放弃预设信号方向，转而通过数据驱动的方式学习特定（环境，骨干网络）组合下的效用方向。其流程分为三个阶段：
*   **探索阶段 (Explore)**：采用信号无关（Signal-agnostic）的随机探索。在交互过程中，以 Bernoulli-$\epsilon$ 的概率随机触发优化器，并记录“执行优化器动作”与“执行基础动作”之间的反事实收益差（Utility $U$）。
*   **推理阶段 (Reason)**：构建候选特征池 $\phi_{cand}$。
    *   **通用特征**：如步骤数、Token 熵、证据数量、可用动作数等。
    *   **LLM 提议特征**：利用 LLM 分析探索数据，自动生成 5 个任务相关的 Python 特征提取函数。
*   **学习阶段 (Learn)**：拟合一个带有 $\ell_1$ 正则化的逻辑回归门控模型。
    *   **公式**：$g(s) = \mathbb{1}[\sigma(w^{*\top} \phi(s) + b^*) > \tau]$。
    *   **方向发现**：通过权重 $w_i$ 的正负号识别特征属性。$w_i > 0$ 代表该特征指示“决策困难”（适合计算）；$w_i < 0$ 代表“干预不宜”（计算可能有害）。

*   **骨干网络**：Qwen3-4B-Instruct, Phi-3.5-mini-Instruct, Llama-3.1-8B-Instruct。
*   **对比基准 (Baselines)**：
    *   **固定方向/预算方法**：CaTS, SEAG, CoRefine, CATTS, AUQ, s1 budget。

*   **评估指标**：成功率 (SR) 和 计算成本 (Cost，以基础 Token 消耗的倍数衡量)。

### 5. 实验数量与充分性
*   **实验规模**：在 6 个环境、3 个骨干网络上进行了完整测试，每个配置运行 3 个随机种子。
*   **消融实验**：非常充分，包括方向反转测试、门控复杂度（线性 vs MLP vs 探针）对比、正则化项对比、LLM 特征贡献分析等。
*   **客观性**：作者不仅展示了 DIAL 的优越性，还详细分析了其在 FEVER 环境下的弱点，并提出了“双源模型”（Type I 不宜干预 vs Type D 决策困难）来解释信号反转的深层原因，具有较强的理论说服力。

### 6. 主要结论与发现
*   **信号语义的易变性**：信号的方向不仅取决于任务，还取决于模型骨干。例如在 FEVER 任务中，熵与效用的相关性在 Phi-3.5 上是负向的，而在 Llama-3.1 上是正向的。
*   **DIAL 的有效性**：在 18 个（环境，骨干）组合中，DIAL 在 16 个组合中取得了最高的成功率，且通常伴随着更低的计算成本。
*   **计算并非总是益处**：在某些状态下（如信息极度匮乏时），额外的 Rollout 会放大错误而非纠正错误。DIAL 能够学习在这些状态下“保持沉默”。

### 7. 优点与亮点
*   ** reframing 问题**：将门控问题从“难度估计”重新定义为“干预效用估计”，揭示了现有方法的结构性缺陷。
*   **轻量化部署**：DIAL 训练完成后，推理时仅需计算几个特征和一个 Sigmoid 函数，几乎不增加额外的推理延迟或 LLM 调用开销。
*   **自动化特征工程**：结合了通用特征和 LLM 自动生成的特征，兼顾了泛化性和任务特定性能。

### 8. 不足与局限
*   **探索成本**：DIAL 需要在部署前进行一轮（约 50 个 Episode）的随机探索来收集数据，虽然是单次投入，但在极高成本的环境下仍需考虑。
*   **FEVER 环境的挑战**：在 FEVER 任务中，DIAL 的表现虽优于基准，但距离“总是触发”的上限仍有差距，说明随机探索可能漏掉了某些关键的效用信号。
*   **静态假设**：目前的 DIAL 主要针对静态环境，虽然文中提到了在线更新的潜力，但在快速漂移的环境下的表现尚未深入探讨。
