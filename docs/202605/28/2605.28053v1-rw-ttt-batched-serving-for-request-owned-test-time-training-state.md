## 摘要
测试时训练在生成过程中通过读取和更新请求自有状态（如快速权重、低秩增量或流式学习器状态）来适应大语言模型。这打破了假设共享静态权重的批处理LLM服务：串行执行正确但缓慢，而朴素的批处理可能破坏请求状态。我们将此问题形式化为读写TTT服务，并提出RW-TTT，它为每个解码步骤标记所有者、版本和读/写效果，仅对兼容阶段进行批处理，并仅向所有者提交更新。在具有八个快速权重InPlace-TTT流的一块GPU上，RW-TTT达到274.61聚合tokens/秒，比顺序服务快9.31倍，在相同内存预算下比每流副本快3.44倍。它在长上下文基准测试RULER上保持行为，并通过所有者/版本检查。

## Abstract
Test-time training (TTT) adapts an LLM during generation by reading and updating request-owned state, such as fast weights, low-rank deltas, or streaming learner state. This breaks batched LLM serving, which assumes shared static weights: serial execution is correct but slow, while naive batching can corrupt request state. We formulate this problem as read-write TTT serving and present RW-TTT , which tags each decode step with its owner, version, and READ/WRITE effect, batches only compatible phases, and commits updates only to the owner. On one GPU with eight fast-weight InPlace-TTT streams, RW-TTT reaches 274.61 aggregate tok/s, 9.31x over sequential serving and 3.44x over per-stream replicas under the same memory budget. It preserves behavior on RULER, a long-context benchmark, and passes owner/version checks.

# RW-TTT: Batched Serving for Request-Owned Test-Time Training State 论文总结

## 1. 核心问题与整体含义

- **研究动机**：测试时训练（Test-Time Training, TTT）让大语言模型在生成过程中动态更新自身状态（如快速权重、低秩增量、流式学习器状态），从而适应特定请求。然而，现有 LLM 服务系统（如 vLLM、S-LoRA）都假设模型参数在整个生成过程中是**共享且只读**的，这导致 TTT 无法被高效地批量服务。
- **核心矛盾**：顺序执行（串行为每个请求逐个生成）虽然保证正确性，但吞吐率极低；而朴素批处理（将多个请求合并执行）会因读写冲突而破坏请求自身的可变状态。
- **整体含义**：需要一种新的服务契约和批处理运行时，能够在保证请求状态隐私、版本正确性和语义一致的前提下，对拥有可变模型状态的请求进行高效批处理。

## 2. 方法论

### 核心思想
- 将每个解码步骤标记为 `READ` 或 `WRITE` 操作，并记录其**所有者（owner）** 和**状态版本（version）**。
- 仅对**兼容阶段**（同为 READ 或同为 WRITE，且后端类型、算子形状等匹配）的请求进行批处理。
- WRITE 步骤更新后的状态仅在成功提交后才对新版本可见，并通过**所有者索引表**确保更新只回到该请求私有的状态槽。

### 关键技术细节
- **请求模型**：每个请求 `r` 包含普通推理状态和可变 TTT 状态 `s_v^r`（版本 v）。READ 操作保留版本不变，WRITE 操作消耗更新证据 `u_r` 并产生新版本 `v+1`。
- **合法批处理条件**：同一个操作组内的所有请求必须具有相同的状态效果（ρ）、后端类型（τ）、算子形状（σ）以及当前有效版本（与系统记录的所有者版本一致）。
- **有限等待预算**：调度器可以将某个 READY 的步骤最多延迟 w 步，以等待更多兼容请求到达，从而增加批处理密度，但不改变单个请求的读写顺序。
- **版本控制和回滚**：对每个所有者保存候选状态和快照，支持 WRITE 组失败后的回滚以及投机执行下的分叉。
- **算法流程**（主文 Algorithm 1）：
  1. 遍历所有活跃请求，获取其下一个步骤（READ/WRITE）及当前版本。
  2. 调用 `LEGAL_GROUPS` 将兼容步骤分组（满足同一效果、后端类型、算子形状和版本匹配）。
  3. 对每组执行 `EXECUTE_OPERATOR_GROUP` 得到输出和状态增量。
  4. 若是 WRITE 组，调用 `COMMIT_VERSIONS` 将新版本提交给对应的所有者。
  5. 更新 KV 缓存和尾部元数据。

## 3. 实验设计

- **模型**：Qwen3-4B 的 In-Place-TTT 快速权重检查点（Feng et al., 2026）。

## 6. 主要结论与发现

1. **核心发现**：RW-TTT 能够在保持请求可变状态语义正确的前提下，大幅恢复批处理吞吐能力。在单 GPU 八流场景下，完整系统达到 **274.61 tok/s**，分别是顺序服务的 **9.31 倍** 和内存匹配副本的 **3.44 倍**。
2. **行为保持**：在 RULER-32K/64K 上，RW-TTT 执行结果与原 In-Place TTT 路径几乎一致（32K: 76.41 vs 76.30；64K: 67.72 vs 67.72）。
3. **契约正确性**：通过所有数值门限和状态压力测试，证明所有者索引、版本提交、回滚机制有效。
4. **瓶颈分析**：吞吐损失的主要原因是无法安全地批处理可变状态，而非单请求 TTT 计算开销（RW-TTT 单请求额外开销仅 1.4%）。
5. **算子层面**：WRITE 侧的状态提交和更新是新的优化热点，而读路径可由标准库高效执行。

## 7. 优点

- **形式化契约**：提出了清晰的服务契约（所有者、版本、READ/WRITE 效果、合法批处理条件），为后续研究提供了理论基础。
- **通用性**：接口设计允许快速权重、低秩增量、流式学习器等多种 TTT 后端共享同一服务运行时。
- **版本管理与回滚**：显式支持状态版本化和故障恢复（回滚、投机执行），增强了鲁棒性。
- **全面的验证**：不仅报告吞吐量，还通过任务分数、数值门限和契约压力测试确认行为不变，避免了仅依赖 token 匹配的片面性。
- **轻量级**：在单 GPU、8 流场景下即可获得显著加速，不需分布式或多卡。
